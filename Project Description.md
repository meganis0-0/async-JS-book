## Глава 0 — Основы HTML, CSS, JS

### 0.1 — Первый скрипт

#### `0.1/index.html`

Минимальная HTML-страница с кнопкой `#myButton`. Подключает `script.js` и `style.css`. Сама по себе ничего не делает — служит оберткой для первого JavaScript-файла.

#### `0.1/script.js`

**Что происходит:**

```js
document.getElementById('myButton').addEventListener('click', function() {
  alert('Кнопка нажата!');
});
```

Первое знакомство с DOM и событиями. Скрипт находит кнопку по `id`, вешает на неё обработчик события `click`, и при нажатии показывает всплывающее окно через `alert`.

**Ключевые концепции:**
- `document.getElementById` — поиск элемента в DOM
- `addEventListener` — подписка на событие
- Колбэк-функция как обработчик события
- `alert` — синхронный диалог браузера

### 0.4 — Часы с тёмной темой

#### `0.4/index.html`

Страница с тремя элементами: `#time` (время), `#date` (дата), `#themeBtn` (кнопка "Ночной режим"). Сама по себе — просто каркас. Вся логика в `script.js`.

#### `0.4/script.js`

**Что происходит:**

Скрипт создаёт живые часы и переключатель темы.

```
1. setInterval(updateClock, 1000) — каждые 1000 мс вызывает функцию обновления
2. updateClock() берёт new Date() и форматирует:
   - time: toLocaleTimeString('ru-RU') → "14:35:07"
   - date: toLocaleDateString('ru-RU') → "28.03.2026"
3. Кнопка темы: classList.toggle('dark') переключает класс,
   classList.contains('dark') проверяет текущее состояние
```

**Ключевые концепции:**
- `setInterval` — повторяющийся таймер (асинхронность!)
- `Date` — работа с датой и временем
- `toLocaleString` — локализованное форматирование
- `classList.toggle` / `classList.contains` — управление CSS-классами
- `textContent` — обновление текста элемента

### 0.5 — DOMContentLoaded и setTimeout

#### `0.5/index.html`

Страница с заголовком "Мой первый сайт с JavaScript" и кнопкой `#myButton`. Скрипт подключён в конце `<body>` (хорошая практика — DOM уже загружен к моменту выполнения).

#### `0.5/script.js`

**Что происходит:**

```
1. DOMContentLoaded — код внутри выполнится только когда весь HTML будет готов
2. Клик по кнопке → alert("Вы нажали кнопку!")
3. setTimeout(..., 2000) — через 2 секунды:
   - текст кнопки сбрасывается
```

**Ключевые концепции:**
- `DOMContentLoaded` — событие готовности DOM (безопасная точка старта)
- `setTimeout` — отложенное выполнение (первое знакомство с асинхронностью)
- `style.backgroundColor` — прямая установка CSS из JavaScript
- Разница между "сейчас" (синхронно) и "через время" (через таймер)

## Глава 1 — Синхронный JS и стек вызовов

---

### 1.1 — Синхронный код

#### `1.1/sync-code.js`

**Что происходит:**

Три функции выстроены в цепочку: `stepOne` → `stepTwo` → `stepThree`. Каждая принимает результат предыдущей и передаёт его в следующую.

```
stepOne() → возвращает результат
stepTwo(результат) → обрабатывает и возвращает
stepThree(результат) → финальный вывод
```

**Цель:** наглядно показать, что синхронный код выполняется строго по порядку, строка за строкой. Никакой магии — пока `stepOne` не завершилась, `stepTwo` не начинается.

**Ключевые концепции:**
- Синхронное выполнение (sequential execution)
- Цепочка вызовов с передачей данных
- `console.log` как инструмент трассировки

---

#### `1.1/blocked.js`

**Что происходит:**

Функция `heavyOperation(name, duration)` создаёт искусственную задержку через пустой цикл:

```js
const start = Date.now();
while (Date.now() - start < duration) {
  // ничего не делаем — просто крутим процессор
}
```

Затем несколько вызовов `heavyOperation` выстроены последовательно.

**Цель:** показать, что "тяжёлая" синхронная операция _буквально останавливает_ всё остальное. Пока работает цикл — браузер не реагирует на клики, не обновляет UI, не выполняет другой код.

**Ключевые концепции:**
- Блокировка основного потока (main thread blocking)
- `Date.now()` для измерения времени
- Busy-wait — антипаттерн синхронной задержки
- Цена синхронных "тяжёлых" операций

---

#### `1.1/sync-problem.js`

**Что происходит:**

Симуляция того, как выглядел бы синхронный сетевой запрос. Используется `heavyOperation` (из `blocked.js`) для имитации "ожидания сети".

```
1. Запрашиваем данные (sync) → heavyOperation блокирует на 2 секунды
2. Только после этого код продолжается
3. Никакого взаимодействия с пользователем в это время невозможно
```

**Цель:** объяснить, _почему_ сетевые запросы должны быть асинхронными. Если бы `fetch` был синхронным — браузер замерзал бы при каждом запросе.

**Ключевые концепции:**
- Почему сеть = асинхронность
- Проблема синхронного I/O
- Мотивация для изучения async/await и промисов

### 1.2 — Стек вызовов

#### `1.2/LIFO.js`

**Что происходит:**

Три функции вызывают друг друга: `firstFunction` → `secondFunction` → `thirdFunction`.

Порядок вывода в консоли:
```
"Начало firstFunction"
"Начало secondFunction"
"Начало thirdFunction"
"Конец thirdFunction"    ← последняя вошла — первая вышла
"Конец secondFunction"
"Конец firstFunction"
```

**Цель:** продемонстрировать принцип LIFO (Last In — First Out) — фундаментальное свойство стека вызовов.

**Ключевые концепции:**
- Стек вызовов (Call Stack)
- LIFO — последним вошёл, первым вышел
- Как JS-движок "помнит" где продолжить выполнение после возврата из функции

---

#### `1.2/stack-ex1.js`

**Что происходит:**

Сценарий заказа в магазине:

```
main()
  └── processOrder(items)
        └── calculateTotal(prices)
              └── multiply(price, quantity)
```

Каждый уровень добавляет свой слой в стек. Логи на входе/выходе каждой функции показывают, как стек растёт и сжимается.

**Цель:** показать стек в реальном бизнес-сценарии, не абстрактный пример.

**Ключевые концепции:**
- Вложенные вызовы функций
- Передача данных между уровнями стека
- Визуализация глубины стека через логи

---

#### `1.2/stack-ex2.js`

**Что происходит:**

Функция `demonstrateStackLevels` содержит внутри себя `innerFunction`, которая содержит `deepestFunction`. Параметры передаются вниз по уровням, результаты — вверх.

**Цель:** показать, как замыкания и вложенные объявления функций взаимодействуют со стеком. Самая глубокая функция имеет доступ к переменным верхних уровней (замыкание).

**Ключевые концепции:**
- Вложенные объявления функций
- Замыкания (closures) — доступ к внешним переменным
- Глубина стека при вложенных вызовах

---

#### `1.2/stack-ex3.js`

**Что происходит:**

Цепочка вызовов `levelOne` → `levelTwo` → `levelThree`. Внутри `levelThree` вызывается `investigateCallStack()`, которая создаёт `new Error()` и читает `error.stack`.

```js
const error = new Error('Stack trace');
console.log(error.stack); // выведет всю цепочку вызовов
```

**Цель:** показать, как получить и прочитать трассировку стека. Это тот самый текст, который видишь в консоли при ошибке.

**Ключевые концепции:**
- `try/catch` — перехват ошибок
- `throw new Error` — ручной выброс ошибки
- `error.stack` — строка с трассировкой
- Чтение stack trace для отладки

---

#### `1.2/stack-ex4.js`

**Что происходит:**

```
processData(data)
  └── calculateAverage(numbers)  ← использует reduce
        └── safeDivision(a, b)   ← бросает ошибку при b === 0
```

Если в данных окажется ситуация деления на ноль — ошибка пройдёт вверх по стеку и будет поймана в `processData` через `try/catch`.

**Цель:** показать распространение (propagation) ошибок вверх по стеку вызовов. Ошибку не обязательно ловить там, где она возникла.

**Ключевые концепции:**
- Распространение ошибок по стеку
- `Array.prototype.reduce`
- Разделение ответственности: функция бросает, вызывающий код ловит
- Защитное программирование (`if (b === 0) throw`)

#### `1.2/exercise-1.js`

**Что происходит:**

Три функции `exerciseA`, `exerciseB`, `exerciseC` вызывают друг друга. Упражнение для самопроверки: _до запуска_ нарисуй стек вызовов и предскажи порядок вывода.

**Цель:** закрепить понимание стека через активное предсказание, а не пассивное чтение.

#### `1.2/exercise-2.js`

**Что происходит:**

```js
console.log(1);
setTimeout(() => console.log(2), 0);
console.log(3);
```

Вывод: `1 → 3 → 2`

Несмотря на задержку `0ms`, колбэк `setTimeout` выполняется _после_ всего синхронного кода.

**Цель:** первый "вау-момент" с асинхронностью. Объясняет, что `setTimeout(fn, 0)` означает не "сейчас", а "как только освободится поток".

**Ключевые концепции:**
- Очередь задач (Task Queue / Macrotask Queue)
- Почему `setTimeout(0)` — не синхронно
- Разница между синхронным стеком и очередью задач

### 1.3 — Блокирующие операции

#### `1.3/block-ex1.js`

**Что происходит:**

Функция `blockingOperation()` работает через busy-wait цикл на 3 секунды. После вызова — сразу же ещё один `console.log`. Второй лог появится _только через 3 секунды_.

**Цель:** ощутить на практике, что значит "заблокировать поток". Ждёшь 3 секунды перед следующей строкой кода.

---

#### `1.3/block-ex2.js`

**Что происходит:**

Функция `calculatePrimes(limit)` находит все простые числа от 2 до `limit` методом пробного деления. Запускается дважды — с маленьким и большим `limit`.

```
calculatePrimes(1000)    → быстро
calculatePrimes(1000000) → несколько секунд
```

**Цель:** показать, что реальные вычисления (не просто пустой цикл) тоже блокируют поток. Алгоритмическая сложность имеет значение.

**Ключевые концепции:**
- Вычислительная сложность алгоритмов
- Блокировка CPU-интенсивным кодом
- Пример с простыми числами как типичная CPU-задача

---

#### `1.3/block-ex3.js`

**Что происходит:**

Создаётся большой массив через `Array.from`, затем над ним выполняются:
1. `map` с тяжёлыми вычислениями и `spread`-операцией
2. `sort` с нетривиальным компаратором (`complexScoringAlgorithm`)

**Цель:** показать, что стандартные методы массивов (map, sort) тоже могут блокировать поток при больших данных. "Функциональный стиль" не означает "неблокирующий".

**Ключевые концепции:**
- Скрытая стоимость `map`/`sort` на больших массивах
- Почему `sort` с кастомным компаратором — потенциально дорого
- Spread-оператор и его стоимость

#### `1.3/block-vs-noblock.js`

**Что происходит:**

Сравниваются два подхода к "обработке нескольких изображений":

**Блокирующий:**
```
обработка 1 → (поток заблокирован) → обработка 2 → ... → готово
```

**Неблокирующий:**
```
запланировать обработку 1 (setTimeout)
запланировать обработку 2 (setTimeout)
...
каждая выполнится по очереди, не блокируя друг друга
```

**Цель:** показать, как `setTimeout` помогает "нарезать" тяжёлую работу на куски и возвращать управление в браузер между ними.

**Ключевые концепции:**
- Chunking (дробление задач)
- `setTimeout` как способ уступить управление потоку
- Разница в пользовательском опыте: зависание vs плавная работа

#### `1.3/perfomance-use.js`

**Что происходит:**

```js
performance.mark('start');
// ... тяжёлый цикл, строящий объекты ...
performance.mark('end');
performance.measure('операция', 'start', 'end');

const measure = performance.getEntriesByName('операция')[0];
if (measure.duration > 50) {
  console.warn('Слишком долго!');
}
```

**Цель:** показать профессиональный инструмент замера производительности — `Performance API`. Порог в 50 мс — это стандарт (одно обновление экрана при 60fps = ~16 мс, 50 мс — заметная задержка).

**Ключевые концепции:**
- `performance.mark` / `performance.measure` — точные временны́е метки
- `performance.getEntriesByName` — получение результатов замера
- 50 мс как порог заметной задержки для пользователя

---

#### `1.3/problem-diagnostic.js`

**Что происходит:**

`measurePerformance(fn)` — обёртка-декоратор: запускает любую функцию, замеряет время через `performance.now()`, предупреждает если > 100 мс.

```js
const start = performance.now();
fn();
const duration = performance.now() - start;
if (duration > 100) console.warn(`Долго: ${duration}мс`);
```

**Цель:** шаблон для диагностики медленных функций. Этот паттерн можно применять к любому коду.

**Ключевые концепции:**
- `performance.now()` — время в миллисекундах с высокой точностью
- Функции высшего порядка (higher-order functions)
- Паттерн "обёртка для замера"

#### `1.3/problem-visualization.html`

**Что происходит:**

Интерактивная страница с двумя кнопками и счётчиком.

- Кнопка "Тяжёлое вычисление" запускает `heavyCalculation()` — цикл на 5 секунд
- Кнопка "Увеличить счётчик" должна работать всегда
- В реальности — пока работает тяжёлый расчёт, UI полностью зависает

**Цель:** _почувствовать_ блокировку UI лично. Нажимаешь кнопку — страница замерзает. Счётчик не реагирует. Именно это происходит в реальных приложениях при синхронном тяжёлом коде.

### 2.1 — Таймеры, колбэки, fetch, промисы

#### `2.1/async-ex1.js`

**Что происходит:**

```js
console.log('Начало');

// Блокирующая версия (закомментирована):
// heavyOperation(2000);

// Неблокирующая версия:
setTimeout(() => {
  console.log('Асинхронная операция завершена');
}, 2000);

console.log('Код продолжает выполняться немедленно');
```

Вывод:
```
Начало
Код продолжает выполняться немедленно
(через 2 секунды) Асинхронная операция завершена
```

**Цель:** ключевой момент понимания асинхронности: `setTimeout` не ждёт. Код после него выполняется сразу.

**Ключевые концепции:**
- Неблокирующее выполнение
- `setTimeout` как планировщик, а не задержка
- Разница: "запустить" vs "дождаться завершения"

#### `2.1/callbacl-ex1.js`

> **Примечание:** в имени файла опечатка — `callbacl` вместо `callback`.

**Что происходит:**

```js
console.log('Синхронный код начался');

setTimeout(function() {
  console.log('Я — колбэк, выполнюсь через 1 секунду');
}, 1000);

console.log('Синхронный код завершился');
```

Вывод: синхронный старт → синхронный конец → через 1с → колбэк.

**Цель:** первый пример колбэка в контексте асинхронности. Функция передаётся как аргумент и вызывается позже.

**Ключевые концепции:**
- Колбэк-функция (callback function)
- Функция как аргумент (first-class functions)
- Порядок выполнения: синхронный код → очередь задач

#### `2.1/timers.js`

**Что происходит:**

```js
console.log('1 - синхронно');

setTimeout(() => console.log('2 - макрозадача 1000мс'), 1000);
setTimeout(() => console.log('3 - макрозадача 0мс'), 0);

Promise.resolve().then(() => console.log('4 - микрозадача'));

console.log('5 - синхронно');
```

Вывод:
```
1 - синхронно
5 - синхронно
4 - микрозадача   ← промис выполнился до setTimeout(0)!
3 - макрозадача 0мс
2 - макрозадача 1000мс
```

**Цель:** объяснить три уровня приоритета в цикле событий: синхронный код → микрозадачи (промисы) → макрозадачи (setTimeout).

**Ключевые концепции:**
- Event Loop — цикл событий
- Microtask Queue — очередь микрозадач (промисы, `queueMicrotask`)
- Macrotask Queue — очередь макрозадач (setTimeout, setInterval)
- Порядок: Стек → Микрозадачи → Одна Макрозадача → Рендер → Повтор

#### `2.1/visualization,js`

> **Примечание:** файл назван с запятой в расширении — опечатка в имени.

**Что происходит:**

Та же демонстрация цикла событий, что и в `timers.js`, но с метками `T+0ms`:

```js
console.log('[T+0ms] Синхронный старт');
setTimeout(() => console.log('[T+1000ms] setTimeout 1000'), 1000);
setTimeout(() => console.log('[T+~0ms] setTimeout 0'), 0);
Promise.resolve().then(() => console.log('[T+0ms] Promise microtask'));
console.log('[T+0ms] Синхронный конец');
```

**Цель:** то же, что и `timers.js`, но с временны́ми метками — помогает понять реальный тайминг выполнения.

#### `2.1/fetch-ex1.js`

**Что происходит:**

```js
console.log('1. Начало');

fetch('https://example.com/api/data')
  .then(response => {
    console.log('3. Получен ответ');
    return response.json();
  })
  .then(data => {
    console.log('4. Данные готовы:', data);
  });

console.log('2. Запрос отправлен, продолжаем...');
```

Вывод: `1 → 2 → (через время сети) 3 → 4`

**Цель:** первый пример реального асинхронного сетевого запроса. `fetch` не блокирует — код идёт дальше, а обработчики `.then` вызываются когда придёт ответ.

**Ключевые концепции:**
- `fetch` — браузерный API для HTTP-запросов
- `Promise` — объект, представляющий будущий результат
- `.then(callback)` — обработчик успешного завершения
- Цепочка промисов (Promise chaining)
- `response.json()` — парсинг JSON из ответа

## Приложение 1 — Справочник CSS

#### `Приложение 1/index.html`

**Что происходит:**

Большой одностраничный HTML-файл — рабочая тетрадь по CSS. Содержит разделы:

**5. Блочная модель (Box Model)**
- `content`, `padding`, `border`, `margin` — все слои с размерами
- `box-sizing: border-box` vs `content-box`
- Встроенные примеры кода в `<pre>`

**6. Типографика**
- `font-family`, `font-size`, `font-weight`, `line-height`
- `text-align`, `text-decoration`, `letter-spacing`

- `color`, `background-color`
- Форматы: именованные, `#hex`, `rgb()`, `rgba()`, `hsl()`
- Прозрачность и альфа-канал

**Цель:** полноценный визуальный справочник. Каждая концепция сопровождается: описанием, CSS-кодом в `<pre>`, и живым примером прямо в браузере. Используется как шпаргалка и для самостоятельных экспериментов.
